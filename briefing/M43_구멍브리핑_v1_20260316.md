## 원인 분석

### 왜 이런 불일치가 생겼나?

#### 🔴 Gap 1 — L1-VH / L1-CHG `draft` 방치

**원인**: VH와 CHG는 LAYER 1 중 가장 먼저 생성된 프레임워크(2026-03-11). 이후 L5-VERIFY에서 Group별 일괄 `finalized` 처리를 했는데, 이때 **`core_theory` type만 대상**으로 UPDATE 쿼리를 날렸고, `framework` type은 조건에 포함되지 않아 누락됨. MSK/HAB/SEX/DIV/MS는 L1-MSK 이후 개별 INSERT 시 처음부터 `finalized`로 저장됐으나, VH/CHG는 초기에 `draft`로 생성된 후 별도 UPDATE가 없었음.

**증거**: CHANGELOG SESSION-20260313-15에서 "D1-001×2, D2-001, D4-001, D8-001 → finalized" 처리 기록 있음. VH/CHG는 언급 없음.

---

#### 🔴 Gap 2 — `m43_research_outputs` analysis 전체 `draft`

**원인**: `research_outputs`는 `analysis` 외에 `case_study`, `field_note`, `hypothesis`, `literature`, `statement`, `rebuttal` 등 **7가지 type**으로 구성됨. 마체리에서 "231개 완료"는 **도메인당 1개의 analysis**를 기준으로 체크했지만, 실제로는 **다종 type이 한 도메인에 여러 개** INSERT된 구조였음.

실제 분포:
- `analysis`: draft 231개 + published 89개 → **142개 unpublished**
- 나머지 6 type: 각 231~232개, 모두 `published`

**핵심 원인**: 배치 INSERT 시 `analysis` type은 `status='draft'`로 생성하고, 이후 개별 `published` 전환을 하도록 설계됐으나 **D1~D5 / 일부 Division은 전환 작업이 생략**됨. 마체리 체크 기준이 "행 수 = 231" 였고 status 확인 없이 체크.

---

#### 🔴 Gap 3 — `m43_sessions` millarche_dh 9개 부족

**원인**: 세션 생성은 Division별 배치로 진행됐고, 최종 카운트 기준이 `SELECT COUNT(*) FROM m43_sessions` = 231개였으나 실제로는:
- `millarche_gh` (Group Head 세션) 12개가 **별도로 존재**
- `millarche_all` 1개 교차 세션 존재
- `millarche_dh` = 235 - 12 - 1 = **222개** (9개 부족)

즉, 231개 카운트 안에 `millarche_dh`가 아닌 세션이 섞여 있었고, 마체리 완료 판정은 총 행 수 기준이었기 때문에 gap 미발견. **어떤 Division의 millarche_dh 9개가 없는지는 추가 조회 필요.**

---

#### 🟡 Gap 4 — `domain_answers` verified=FALSE 16개

**원인**: 질문/답변 생성이 두 세션에 걸쳐 진행됨 (SESSION-20260314-1: 1,232개, SESSION-20260314-2: 교차 90개 추가). 일부 답변이 INSERT 시 기본값 `verified=FALSE`로 들어갔고, 배치 UPDATE에서 누락된 행이 16개 남음. current_state.md에서 "미답변 0, verified=TRUE 전체"로 기록했으나 실제와 불일치.

## 🔴 Gap별 수정 방법

### Gap 1: VH/CHG → finalized 전환
```sql
UPDATE m43_domain_theories
SET status = 'finalized', updated_at = now()
WHERE id IN (
  '3d5dda7a-eb19-415a-bcb7-eddb69d38bf6',  -- VH
  '9105d82c-4eb1-47bf-bbdc-5cf03bfb9034'   -- CHG
);
```

### Gap 2: analysis published 전환 (142개)
```sql
-- analysis draft 상태 전체 published로 전환 (의도적 draft면 제외 확인 필요)
UPDATE m43_research_outputs
SET status = 'published', updated_at = now()
WHERE type = 'analysis' AND status = 'draft';
```
> ⚠️ 실행 전: 의도적으로 draft로 둔 항목 없는지 확인할 것

### Gap 3: millarche_dh 누락 9개 식별
```sql
-- millarche_dh가 없는 도메인 찾기
SELECT d.code, d.name
FROM m43_domains d
WHERE NOT EXISTS (
  SELECT 1 FROM m43_sessions s
  WHERE s.domain_ids @> ARRAY[d.id]::uuid[]
  AND s.type = 'millarche_dh'
)
ORDER BY d.code;
```
> 식별 후 해당 도메인의 millarche_dh 세션 신규 생성 필요

### Gap 4: verified=FALSE → TRUE 전환
```sql
UPDATE m43_domain_answers
SET verified = TRUE, updated_at = now()
WHERE verified = FALSE;
```
> 단, 실제로 답변 품질 확인 후 진행 권장 (16개이므로 수동 검토 가능)

| 문제 패턴 | 원인 | 재발 방지 |

## Gap 3 재분석 — 오진 판정

**최초 진단**: millarche_dh 9개 부족
**재분석 결과**: 실제 설계대로임. 오진.

- 각 Division `-001` 도메인 12개는 `millarche_gh`(밀라르쉬 × Group Head) 타입으로 커버
- `millarche_dh`가 없는 도메인은 없음 (세션 없는 도메인 = 0개 확인)
- 세션 총 235개 = millarche_dh 222개 + millarche_gh 12개 + millarche_all 1개
- 222 + 12 = 234개가 231개 도메인 커버 (일부 도메인은 두 타입 모두 있을 수 있음)
- **실제 누락 없음**
