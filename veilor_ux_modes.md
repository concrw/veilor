# VEILOR UX 모드 설계 문서

> 작성일: 2026-04-13  
> 목적: 베일러 3가지 UX 패러다임 모드 설계. 구현의 기반 문서.

---

## 개요

> 최종 확정: 2026-04-14

베일러는 동일한 데이터 위에 **3가지 UX 패러다임**을 제공한다.

| 모드 | 이름 | 한 줄 정의 | 상태 |
|---|---|---|---|
| 1 | **플로우모드 (오리지널)** | 홈의 초기 디자인 분위기를 모든 페이지에서 일관되게 유지 | 기준 확정 — 전 페이지 적용 필요 |
| 2 | **음성모드** | 음성 중심 인터랙션 — 화면 UI 기준 미정의 | 설계 필요 |
| 3 | **시스템모드 (가제)** | 경쟁사 UI/UX 벤치마킹 기반 — 색상이 아닌 UX 구조 자체를 다르게 | 경쟁사 조사 후 설계 |

**공통 원칙**
- DB 쿼리 추가 없음. 같은 데이터를 다르게 렌더링.
- 모드 설정값은 `localStorage` 저장. 서버 부하 없음.
- lazy() + Suspense로 사용하는 모드만 번들 로드.
- 모드 전환은 항상 유저 동의 후 진행. 강제 전환 없음.

> **⚠ 이전 문서의 "클리어 모드"와 "루틴 모드" 명칭은 폐기.**  
> 시스템모드 설계 시 경쟁사 조사 결과에 따라 재정의.

## 경쟁사 분석 요약 (설계 근거)

### 포지셔닝 맵

```
                    기능적 (데이터·성과)  +5
                              │
          BetterUp(+3,+4)     │      Daylio(+4,+3)
                              │
  수동적(-5) ─────────────────┼───────────────────── 능동적(+5)
                    Wysa(0,+1)│  Woebot(-1,+2)
                Reflectly(+3,-1)│
  Calm(-4,-4)  Headspace(-3,-2)│
                              │
                    감성적 (분위기·공감)  -5

  베일러 오리지널:  (+1,  0)  ← 현재
  베일러 클리어:    (+4, +3)  ← 목표
  베일러 루틴:      (+2, +4)  ← 목표
  🔵 블루오션:      (+4, -2)  ← 능동적 기록 + 감성적 경험 → 오리지널이 가장 근접
```

### 리텐션 핵심 인사이트

| 구간 | 최강 전략 | 근거 앱 |
|---|---|---|
| Day 1~7 | 극저마찰 진입 | Daylio (Day-7 리텐션 28~35%) |
| Day 8~30 | 구조화 + 시간 표시 | Headspace |
| Day 30+ | 코치 예약 강제성 | BetterUp (Day-30 리텐션 22~28%) |
| 전 구간 | 따뜻한 피드백 톤 | Calm + Woebot |
| 장기 | AI 자동 인사이트 | Reflectly |

## 모드 1 — 플로우모드 (오리지널)

### 1. 정체성

**한 줄 정의**
"말하지 않아도 알 것 같은 존재, 엠버가 먼저 손 내미는 공간."

**타깃 유저 페르소나**

| 페르소나 | 설명 |
|---|---|
| **밤의 털어내기형** | 하루 끝 감정을 처리하고 싶지만 민폐 주기 싫다. 엠버와의 대화로 하루를 마무리한다. |
| **감정 언어 약자형** | "그냥 이상해"를 출발점으로 삼는다. 구조화된 질문보다 흘러가는 대화가 편하다. |
| **반복 감정 처리형** | 비슷한 상황에서 같은 감정이 반복된다는 걸 어렴풋이 안다. 우선 쏟아내고 싶다. |

**핵심 감정 경험 목표**
- 판단받지 않는다는 안도감
- "이끌려간다"는 수동적 편안함
- 대화 후 "조금 가벼워진" 감각

---

### 2. 디자인 언어

**색상 시스템** — 기존 토큰 100% 유지

```
배경:       #1C1917  (stone-950)
서피스:     #292524  (stone-800)
포인트:     #D4A574  (amber-gold)
텍스트 주: #F5F5F4
텍스트 보: #A8A29E
경계:       #44403C
```

> 앰버 골드 외 채도 있는 컬러 추가 금지.

**타이포그래피**
- 엠버 발화: Pretendard 400, line-height 1.7 — 숨쉬는 여백
- 시스템 넛지: 소문자, letter-spacing +0.03em — 속삭이듯

**애니메이션**
- 전환: `ease-out` 280ms. 느리게 열리고 빠르게 닫힌다.
- 엠버 응답: 단어 단위 fade-in (50ms 간격). 타이핑 효과 아님.
- 감정 처리 완료: 화면 전체 0.4초 dim → 복귀. "내려놓음" 은유.

**레이아웃**
- 단일 컬럼, 현재 뷰포트 내 완결
- 하단 입력창 고정. 유저 행동 진입점이 사라지지 않음.

---

### 3. 인터랙션 패턴

**진입 흐름**

```
앱 오픈
  └─ 엠버 인사 (시간대 개인화)
       "오늘 하루 어땠어? 천천히 얘기해도 돼."
  └─ 유저: 텍스트 or 음성
  └─ 엠버: 반영형 응답 (해결책 제시 금지)
       "그게 많이 쌓였겠다."
  └─ 3~5턴 후: 감정 요약 카드
       "오늘 네가 많이 지쳤던 거, 나는 알았어."
  └─ 종료 or 다음 탭 넛지
```

**넛지 방식**
직접 CTA 버튼 금지. 엠버 대화 말미에 자연 삽입.

> "이 감정, 이번이 처음은 아닌 것 같아. 들여다볼까?" → [응, 볼게] [아직은 괜찮아]

선택지 항상 2개. 거절 옵션 항상 제공.

---

### 4. 탭별 UX

**Vent** — 구조 없음. 주제를 바꿔도 OK.
- 핵심 인터랙션: 음성 입력 중 실시간 감정 파형. 말이 끝나면 파형이 천천히 가라앉음.

**Dig** — 분석 용어 금지. "패턴" → "자주 느끼는 것"으로 표현.
- 엠버가 찾아낸 것 카드: "3주째 월요일 저녁에 비슷한 감정이 나와. 알고 있었어?"

**Get** — 추천 리스트 아님. 엠버가 1가지만 건넨다.
- "오늘 기분에 이게 맞을 것 같아서 골라봤어." — 큐레이션 이유 항상 포함.

**Set** — 유저가 목표 입력 안 함. 엠버가 Vent/Dig 기반으로 제안.
- 수치 대신 관계적 언어: "지난주보다 조금 나아진 것 같은데, 느꼈어?"

**Me** — 설정 페이지 아님. "나에 대한 기록" 공간.
- 상단: 엠버가 정의한 "이번 달의 나" 요약 문장
- 중단: 감정 히트맵 (amber 농도)

---

### 5. 리텐션 루프

**재방문 트리거**
- 푸시: 엠버 목소리 톤, 저녁 9~11시 기본
  > "오늘 어땠어? 나 여기 있어."

**스트릭/보상**
- 수치형 스트릭 없음. 엠버의 관계 언어로 대체.
  > "우리 얘기한 지 7일 됐어. 꽤 됐다."
- 보상 = 엠버의 응답 개인화 깊이 상승

**장기 유지**
- 30일: "엠버 편지" — 한 달 감정 여정을 엠버 시점으로 서술
- 계절별: 산문 형태 감정 리포트 (수치 그래프 없음)

---

### 6. 모드 전환 진입점

| 상황 | 전환 제안 |
|---|---|
| 안정기 진입, 감정 기복 낮아짐 | → 시스템모드 제안 |
| 음성으로 주로 사용하고 싶다는 표현 | → 음성모드 제안 |

엠버 대화 말미에 삽입. 하드 팝업 금지.
> "요즘 비슷한 얘기를 하는 것 같아. 조금 다른 방식으로 볼 수 있을 것 같아. 해볼까?"

## 모드 2 — 음성모드

> 상태: **설계 필요** (2026-04-14 기준 화면 UI 기준 없음)

### 정의

텍스트 입력 없이 **음성만으로** 베일러의 모든 핵심 기능을 사용할 수 있는 모드.  
화면은 최소화되고, 음성이 주된 인터랙션 수단이 된다.

### 설계가 필요한 질문들

- 음성 입력 중 화면에 무엇이 보여야 하는가? (파형? 텍스트 변환? 아무것도 없음?)
- 음성 응답도 TTS로 나오는가, 텍스트로만 표시되는가?
- 탭 네비게이션은 어떻게 처리하는가? (음성 명령? 터치 유지?)
- 화면을 보지 않고도 사용 가능한 수준인가?
- 운전 중, 산책 중 등 핸즈프리 시나리오를 커버하는가?

### 벤치마킹 대상 (조사 필요)

- Apple Siri / Google Assistant — 음성 전용 UI 패턴
- Alexa 화면 인터페이스 — 음성+최소 화면 조합
- Otter.ai — 실시간 음성→텍스트 변환 UX
- 기존 VEILOR Vent 탭 음성 입력 — 현재 구현된 수준 파악 후 확장

### 현재 구현 상태

- Vent 탭에 음성 입력 기능 일부 존재 (`invokeHeldChat`)
- 독립된 "음성모드" 화면 없음
- TTS(텍스트→음성 출력) 미구현

## 모드 3 — 시스템모드 (가제)

> 상태: **경쟁사 UI/UX 조사 후 설계** (2026-04-14 기준 미정)

### 설계 원칙

- 색상만 바꾸는 것이 아님. UX 구조 자체를 다르게 설계.
- 경쟁사들이 검증한 인터랙션 패턴을 베일러에 맞게 적용.
- 플로우모드(오리지널)와 명확하게 다른 경험을 제공해야 함.

### 조사해야 할 경쟁사

| 앱 | 조사 포인트 |
|---|---|
| Daylio | 극저마찰 체크인, 감정 태그 UI |
| Headspace | 코스/세션 구조, 진척 시각화 |
| BetterUp | 코치 연결, 목표 추적 UX |
| Reflectly | AI 자동 인사이트, 저널 플로우 |
| Woebot | 대화형 CBT 구조 |
| Calm | 온보딩, 컨텐츠 소비 흐름 |
| Wysa | 감정 체크인 → 콘텐츠 연결 방식 |

### 조사 후 결정할 것

- 모드 최종 이름
- 홈 화면 구조 (대시보드형? 코스형? 피드형?)
- 탭별 UX가 플로우모드와 어떻게 달라지는가
- 타깃 유저 페르소나
- 디자인 언어 (색상, 레이아웃, 타이포그래피)

> 이하 구 "클리어 모드"와 "루틴 모드" 상세 내용은 참고용으로만 유지. 확정 설계 아님.

**핵심 감정 경험 목표**
- "오늘 내가 한 일이 보인다" → 효능감
- "내 패턴이 읽힌다" → 자기이해
- "다음에 뭘 하면 되는지 안다" → 방향감

---

### 2. 디자인 언어

**색상 시스템** — 기존 토큰 기반, 쿨톤으로 전환

| 역할 | 오리지널 | 클리어 |
|---|---|---|
| 배경 | #1C1917 (웜 다크) | #111318 (쿨 다크) |
| 포인트 | #D4A574 앰버 | #4AAEFF 클리어블루 |
| 서브 | #7BA8C4 프로스트 | #34C48B 민트 (완료·진척) |
| 카드 경계 | 낮은 대비 | 선명한 경계, 높은 대비 |
| 진척 바 | 없음 | #4AAEFF → #34C48B 그라디언트 |

**타이포그래피**
- 제목: Pretendard 700, 계층 명확화
- 숫자/지표: Tabular nums, 우측 정렬
- 감성 문장보다 동사형 짧은 문장: "완료" / "오늘 3/5"

**레이아웃**
- 상단 고정: 오늘의 진척 요약 카드
- 그리드 기반 2열 배치 허용
- 완료 항목: 체크 + 그레이아웃 (쌓이는 시각화)

**애니메이션**
- 완료 시: 체크 스탬프 + 프로그레스바 0.3s 채워짐
- 전환: 오른쪽 슬라이드 (스텝 진행감)

---

### 3. 인터랙션 패턴

**진입 흐름**

```
앱 열기
  └─ 오늘의 클리어 카드 (상단 고정)
       "오늘 할 것 3가지 · 완료 1/3"
         ├─ [감정 체크인 →] — 미완료 (파란 테두리)
         ├─ [Dig 세션 →]   — 완료 ✓ (그린 체크)
         └─ [저널 한 줄 →] — 미완료
```

**AI 가이드 질문 (Reflectly 스타일)**
단계형 질문 카드 — 한 번에 한 질문:

```
1단계: "오늘 가장 무거운 감정은?"
        [분노 / 불안 / 무기력 / 기타]
         ↓
2단계: "그 감정, 언제부터였어?"
        [오늘 아침 / 며칠째 / 모르겠어]
         ↓
3단계: "지금 가장 하고 싶은 건?"
        [털어내기 / 파고들기 / 그냥 기록]
         ↓ 해당 탭으로 자동 이동
```

**진척도 표시**
- 탭 아이콘 위 뱃지: 오늘 완료 횟수
- 주간 히트맵: 7일 × 5탭 그리드
- 세션 종료 시: "오늘 3번째 기록 완료 · 이번 주 +12%" 팝업

---

### 4. 탭별 UX

**Vent** — 감정 태그 선택 + 강도 슬라이더 + 한 줄 맥락
- 진척 시각화: "이번 달 Vent 8회 · 평균 강도 6.2 → 5.1 감소중"

**Dig** — 코스형 세션 (1회 15분, 5스텝)
- 코스 진행 바 + 완료 뱃지 컬렉션

**Get** — 큐레이션 플레이리스트 구조
- 주간 콘텐츠 소화율 % 표시

**Set** — OKR 스타일 목표 추적
- 목표별 프로그레스바 + AI 달성 예측
- "이 속도면 달성 가능 ✓"

**Me** — 대시보드형 자기 분석 허브
```
┌─ 이번 달 요약 ─────────────────┐
│ 총 기록 23회  평균 감정강도 5.4 │
│ 가장 많은 감정: 불안(9회)       │
│ 최장 스트릭: 7일               │
└────────────────────────────────┘
```

---

### 5. 리텐션 루프

**주간 미션 구조**

```
1주차: "기초 체크인" — 매일 감정 체크인 5회
2주차: "패턴 발견"  — Dig 코스 1개 완료
3주차: "루틴 정착"  — Set 목표 70% 달성
4주차: "월간 리포트" — 자동 생성
```

**완료 보상**
- 즉각: 체크 애니메이션 + 클리어 포인트
- 주간: "이번 주 완벽 클리어 ✦" 뱃지

## 커뮤니티 — 탭 내장형 설계 방향

> 작성일: 2026-04-14  
> 결정: `/home/community` 별도 페이지 라우트 방식 **폐기**. 커뮤니티는 각 탭에 녹아드는 형태로 재설계.

### 핵심 원칙

각 탭에서 **"지금 같은 경험을 하고 있는 사람들의 목소리"** 가 자연스럽게 보여야 한다.  
유저가 커뮤니티를 찾아가는 것이 아니라, 탭 안에서 공동체를 만나는 구조.

### 탭별 커뮤니티 역할

| 탭 | 보여줄 목소리 | 맥락 |
|---|---|---|
| **Vent** | 같은 감정을 털어놓은 사람들의 익명 한 줄 | "지금 불안한 사람이 142명 있어요" — 혼자가 아님을 느끼게 |
| **Dig** | 같은 패턴을 가진 사람들이 발견한 것 | "이 패턴을 가진 사람들이 가장 많이 발견한 것" |
| **Get** | 같은 마스크 유형 사람들의 성장 경험 | "같은 가면을 가진 사람들이 쓴 이야기" |
| **Set** | 같은 목표를 실천 중인 사람들의 기록 | "이 키워드로 실천하는 사람 23명" |
| **Me** | 나와 비슷한 V-Axis 프로필을 가진 사람들 | 연결 제안, 코호트 |

### 구현 방향

- 각 탭 하단 또는 AI 응답 이후 섹션에 **"다른 사람들은"** 블록으로 삽입
- 익명 처리 기본값. 실명 선택 가능.
- `community_posts` 테이블의 `category` 컬럼으로 탭별 필터링
- 가상 유저 데이터(`virtualUsers`)를 초기 콘텐츠로 활용 (실 데이터 부족 시)
- 게시/댓글 기능은 탭 내 인라인으로 (별도 페이지 이동 없이)

### 현재 상태 (미구현)

- `CommunityPage.tsx`: 별도 라우트로만 존재, 바텀네비 진입점 없음 → **향후 각 탭 컴포넌트로 분해 예정**
- `DiscussionBoard`, `LearningMateCard`, `CohortCard`, `PartnerCodetalk`: 각 탭에 맞는 컴포넌트로 재배치 예정
- `/home/community` 라우트: 유지하되 내부 링크로만 사용 (직접 진입 용도)

## DM — 진입 구조 설계 방향

> 작성일: 2026-04-14

### 현황

`/home/dm` (`DmPage.tsx`) 은 유저 간 1:1 채팅 기능.  
현재 바텀네비 탭 없음. Me 탭의 DM 버튼도 "준비 중" 상태. 사실상 진입 불가.

### 설계 방향

인스타그램의 메시지 버튼과 동일한 UX.

```
유저 프로필 페이지
  └─ [메시지 보내기] 버튼
       └─ 동의 확인 (consent_a / consent_b)
            └─ DmPage (채팅방)
```

- DM은 독립 탭이 아님. 항상 **특정 유저의 프로필 페이지에서 진입**
- 유저 프로필 페이지가 먼저 만들어져야 DM 진입점이 생김
- 유저 프로필 페이지: Me 탭 "내 사람들" 섹션 또는 커뮤니티 게시글 작성자 터치 시 진입
- 메시지 버튼은 상대방 프로필에만 표시 (본인 프로필에는 없음)
- 양쪽 동의 후 채팅방 생성 (`consent_a`, `consent_b` — 이미 DB 구조에 반영됨)
- 메시지 안전성 필터 Edge Function (`dm-message-filter`) 이미 연동됨

### 구현 선행 조건

1. 유저 프로필 페이지 (`/users/:userId`) 라우트 및 페이지 생성
2. 프로필 페이지에 [메시지 보내기] 버튼 추가
3. DmPage를 해당 유저 ID와 함께 열도록 라우트 파라미터 처리

## 웹 반응형 레이아웃 — 해야 할 작업 목록

> 작성일: 2026-04-14  
> 문제: 전체 앱이 모바일 단일 컬럼(`max-w-sm`) 기준으로만 설계됨. 웹 브라우저 넓은 화면에서 양옆이 텅 비거나 레이아웃이 어색함.

### 현황 진단

| 문제 | 해당 파일 |
|---|---|
| 모든 페이지 콘텐츠가 `max-w-sm mx-auto` (384px) 고정 | 홈 전체 페이지, 온보딩, 인증 |
| 바텀네비가 `fixed bottom-0 inset-x-0` — 화면 전체 너비로 뻗음 | `HomeLayout.tsx` |
| 사이드바 없음 — 넓은 화면에서 네비게이션 구조 없음 | 전체 |
| 배경이 단색(`#1C1917`)으로만 채워짐 — 넓은 화면에서 좌우 여백이 그냥 어두운 벽 | `HomeLayout.tsx` |

### 설계 방향

**기본 원칙: 모바일 우선(Mobile First), 웹에서는 앱이 중앙에 떠 있는 형태**

```
[  웹 브라우저 넓은 화면  ]
┌────────────────────────────────────────┐
│  좌측 여백   │  앱 콘텐츠 영역  │  우측 여백  │
│  (배경/장식) │  max-w-md~lg    │  (배경/장식) │
└────────────────────────────────────────┘
```

**브레이크포인트별 레이아웃**

| 화면 | 레이아웃 |
|---|---|
| `< 640px` (모바일) | 현재와 동일. 전체 너비. 바텀네비. |
| `640px ~ 1024px` (태블릿) | 콘텐츠 `max-w-lg`, 바텀네비 유지 |
| `> 1024px` (데스크톱) | 좌측 사이드바 네비게이션 + 콘텐츠 `max-w-2xl`. 바텀네비 숨김. |

**데스크톱 사이드바 구성**
- 좌측 고정 사이드바: VEILOR 로고 + 탭 메뉴 (Vent / Dig / Get / Set / Me)
- 현재 바텀네비의 탭과 동일한 항목을 세로로 배치
- 너비: 약 200px

**콘텐츠 영역 너비**
- 현재 `max-w-sm` (384px) → `max-w-sm md:max-w-lg lg:max-w-2xl` 로 단계적 확대
- 일부 페이지(Get, Me)는 넓은 화면에서 2컬럼 그리드 레이아웃 적용 가능

**배경 처리**
- 좌우 여백은 동일한 다크 배경색 유지 (단, 미묘한 패턴이나 그라디언트로 "벽" 느낌 완화)
- 또는 앱 영역에 미세한 테두리/그림자로 경계 표시

### 구현 선행 조건

1. `HomeLayout.tsx` — 데스크톱에서 사이드바 + 콘텐츠 2단 구조로 분기
2. 바텀네비 — `lg:hidden` 으로 데스크톱에서 숨김
3. 각 페이지 `max-w-sm` → 반응형 클래스로 교체
4. 온보딩/인증 페이지 — 중앙 카드 형태로 유지 (이미 `mx-auto` 구조, 상대적으로 문제 적음)
